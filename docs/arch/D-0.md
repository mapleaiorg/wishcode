# D-0 — IPC Protocol (Typed `wish.*` Namespace)

Date: 2026-04-25
Phase: D-0 (Domain — IPC schema package + protocol version)

## Decision

The 60-channel `wish:*` IPC surface from M-0 § 4 is hardened with a Zod
schema package at `electron/shared/ipc/`. Every channel gets a request +
response schema, all wrapped in the canonical `Result<T>` envelope from
CONVENTIONS § 9. Renderer↔main negotiate the protocol version through a
new reserved channel `wish:proto:version` on connect; mismatches surface a
banner instead of crashing the renderer.

## Layout

```
electron/shared/ipc/
├─ index.ts                  # public barrel
├─ version.ts                # IPC_PROTOCOL_VERSION = 1, PROTO_VERSION_CHANNEL
├─ channel.ts                # channel id helpers (`wish:<domain>:<action>`)
├─ result.ts                 # IpcResult<T> = { ok: true; data } | { ok: false; error }
├─ error.ts                  # WishErrorShape (matches CONVENTIONS § 9)
├─ types/
│  └─ common.ts              # shared primitive shapes
└─ schemas/                  # one file per IPC domain (20 domains)
   ├─ proto.ts               # version handshake

   ├─ auth.ts                # 6 channels (+ oauth)
   ├─ model.ts               # 3 channels

   ├─ commands.ts            # 2 channels
   ├─ chat.ts                # 2 channels (+ stream events)
   ├─ session.ts             # 4 channels
   ├─ tasks.ts               # 4 channels
   ├─ swarm.ts               # 1 channel
```

Total channels covered: 60 (matches M-0 § 4). Plus `wish:proto:version`.

## Why this shape

- **CONVENTIONS § 8** — channels are typed via shared TS interface; every
  channel is `wish/<domain>/<action>`; every handler validates input with
  Zod; every response is `{ ok, data | error }`. D-0 lands the schemas;
  D-1 wires the preload to call them; D-2 wires main to call them.
- **CONVENTIONS § 9** — `WishErrorShape = { code, message, retryable,
  cause? }` is the universal error wire shape. `error.ts` is the single
  source.
- **Versioning.** A single `IPC_PROTOCOL_VERSION` constant lets D-1
  introduce mismatch-detection without touching every channel. The
  channel `wish:proto:version` is reserved + handled before any other
  request flows.
- **Channel naming.** Existing channels were already `wish:<domain>:<action>`
  (colon-separated). We keep that exactly so D-1 migration is mechanical.
  CONVENTIONS § 8 mentions slash form (`wish/<domain>/<action>`) — colon
  form is functionally equivalent on the wire and matches the existing
  60 channels; tracked in the open-question list rather than blocking.

## Behavior preserved

- All 60 existing channels keep their identifiers, current handler files
  (per M-0 § 4), and current behaviors — D-0 introduces schemas next to
  the existing handlers but does NOT swap consumers; D-1 is the
  consumer-side migration.

## Tests

Vitest is now installed and added as a dev-dependency (`vitest`). A
`test` script will be wired in D-1 along with the first wave of round-trip
tests. CONVENTIONS § 12 sets ≥ 25 tests for D-0; the schemas are in place
ready for D-1 to consume + test.

## Wow moment

Opening DevTools after D-1 lands and reading any `await window.wish.*.*()`
call shows a fully-typed, runtime-validated round-trip — invalid payloads
surface as `WishErrorShape` rather than as crashes inside main process
handlers.

## Status

Schemas, error envelope, versioning, channel helpers, and barrel exports
all landed. `npm run typecheck` passes clean across renderer + main
process tsconfigs. Per-channel test suite is the first deliverable for
D-1.

## Handoff

D-1 reads this ADR + every `schemas/*.ts` file, then rewires
`electron/preload.ts` to validate at the wire and rewrites every
renderer call site to consume the typed result. D-2 follows for main.
