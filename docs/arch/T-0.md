# T-0 — Task vs Job model

Date: 2026-04-26
Phase: T-0 (Task / Job — types + storage primitives)

## Decision

Tasks (user-facing intent) and Jobs (executable graph nodes) are
separate first-class types backed by a single `TaskStore`. Lives at
`electron/native/tasks/v2/` so the legacy flat `tasks/manager.ts`
keeps working until D-3 migrates the IPC.

## Type model

```
Task                 — user intent. One row per ask.
  status: queued | running | paused | succeeded | failed | cancelled
  origin: chat | agent | cron | ipc | cell | system
  bindings: { workspaceId?, sessionId?, agentId?, teamId? }

Job                  — executable node. Many per Task.
  status: pending | ready | running | succeeded | failed | cancelled | skipped
  kind: shell | agent | tool | fetch | noop | <Cell-supplied>
  dependencies: jobId[]   (cycle-checked at insert; same-task only)
  attempts / maxAttempts  (retry semantics owned by the runner)
  state: opaque scratch    (resume support)
```

## Forward-only transition graph

```
Task:   queued → running → { paused | succeeded | failed | cancelled }
                running ↔ paused
        terminal: succeeded | failed | cancelled (frozen)

Job:    pending → ready → running → { succeeded | failed | cancelled }
        pending → cancelled | skipped (e.g. cascadeSkip)
        ready → cancelled | skipped
        terminal: succeeded | failed | cancelled | skipped (frozen)
```

`InMemoryTaskStore` validates every transition; illegal edges throw.

## Job graph mechanics

- `addJob` with no deps lands `ready`; with deps lands `pending`.
- `setJobStatus(j, 'succeeded')` promotes any pending dependent
  whose deps are all `succeeded` to `ready`.
- `cascadeSkip(failedJobId)` walks the reverse-dependency tree and
  marks every still-active descendant `skipped`.
- Dependencies are cycle-rejected at insert (cross-task deps are
  forbidden — every dep must belong to the same Task).
- `readyJobs(taskId)` is what the runner polls.

## Wow moment

A 5-line scenario in the tests proves the graph: create a task, add
A → B → C dependency chain, run A → succeeded → B promotes to ready;
fail A → cascadeSkip drops B and C in one call.

## Handoff

T-1 reads this ADR + the contract; the in-memory store is the
reference for the fs implementation. A-2's agent runtime can drive
Tasks by calling `createTask(...)` and adding `kind: 'tool'` jobs.
C-1 (CoAgent task runtime) shares the same model with a Hermon mirror.
