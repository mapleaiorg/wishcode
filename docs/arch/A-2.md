## The loop, plain

```
while turns < max:
  ev_stream = provider.chat(req with current messages, tools=tools.definitions())
  for each canonical event:
    onEvent(ev)
    if content.completed       => append block to assistant message
    if tool_call.completed     => append tool_use block + queue invocation
    if response.error          => terminal=error
    if response.completed      => terminal iff no tool_calls
  push assistant message
  if terminal: break
  if no tool calls: break (graceful)
  for each invocation:
    output = tools.invoke(invocation, signal)   (or {error} if it throws)
    push canonical {kind:'tool_result', toolUseId, output, isError}
  push tool message
if turns >= max: stopReason = 'max_turns'
```

## Stop reasons

- `completed` — model said done without further tool calls.
- `max_turns` — the loop ran out of iterations.
- `cancelled` — `signal.aborted` observed at a turn boundary.
- `error` — the stream yielded `response.error`.
- `unsupported_tool` — model called a tool but no dispatcher was
  provided.

## Wow moment

`runtime.test.ts` proves the whole loop with a deterministic scripted
provider. Highlights from the 9 tests:
- single-turn completion
- two-turn tool round-trip (`tool_call.completed` →
  `InMemoryToolDispatcher.invoke()` → next turn yields text)
- thrown handler becomes `tool_result{isError:true}` carrying the
  error message
- `unsupported_tool` short-circuit
- `max_turns` cap via a "loop forever" scripted adapter
- `response.error` → `stopReason='error'`
- pre-aborted signal → `stopReason='cancelled'`, `turns=0`
- `onEvent` sees the canonical event sequence live during the run

## Tradeoffs

- The loop is intentionally minimal. Memory adapters (Mem-2),
  knowledge retrieval (K-2), provenance writes (K-3), and
  capability-checked tool dispatch (D-6) all bolt onto the runtime
  through the `ToolDispatcher` + `onEvent` seams without changing
  the loop's shape.
- Canonical-only — the runtime never reaches into provider payload
  shapes. A-1's `ProviderRuntime.chat()` is the producer of every
  event the runtime sees; that boundary is enforced by Zod-validation
  in A-1.
- Tool errors are first-class data, not exceptions: the loop never
  throws, so the host can persist a complete transcript even when
  one tool fails mid-run.

## Handoff

D-2 (main-process IPC handlers) instantiates one `AgentRuntime` per
chat session and feeds streaming events back to the renderer via the
existing `wish:chat:*` event topics. A-3 (provider Cells) replaces the
test `EchoAdapter` with real Anthropic/OpenAI/xAI/Gemini/Ollama/Hermon
adapters — all conform to the same `ProviderAdapter` contract, so the
agent loop is unaffected.
