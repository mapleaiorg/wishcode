# D-6 — Capability Gate

Date: 2026-04-26
Phase: D-6 (Domain — capability broker)

## Decision

Every privileged action — file read/write, git mutate, process spawn,
terminal attach, network fetch, secret read, connector access, CoAgent
mutate — passes through `CapabilityBroker.require(...)` (or a
non-throwing `check(...)` for permissive paths). The broker is the
renderer/main-side gate; defense-in-depth lives in `wishd-capability`
(W-6) on the trusted-runtime side; org-policy gating layers on top via
H-7.

## Surface

```typescript
broker.grant({ subject, kind, constraints?, expiresAt?, reason? })
broker.revoke(grantId)
broker.revokeSubject(subject)         // session end / Cell unload
broker.listFor(subject)               // active grants only (expired filtered)
broker.check({ subject, kind, resource? })   // non-throwing
broker.require(...)                          // throws CapabilityDenied
```

New kinds require an IPC protocol bump (D-0 contract).

## Constraint matching

A grant carries optional `CapabilityConstraints`; absent fields are
wildcards. Concrete constraints:

- `pathPrefixes` — request `path` must equal a prefix or start with
  `prefix/`. Path absent + prefixes set ⇒ deny.
- `hosts` — exact host match (no wildcards yet — H-7 will add policy).
- `providers`, `connectors`, `workspaceIds`, `taskIds` — exact list
  membership, with the same "absent value" deny semantics as paths.

Denial reasons surface a stable enum: `no_grant`, `expired`,
`path_outside_grant`, `host_outside_grant`, `provider_outside_grant`,
`connector_outside_grant`, `workspace_outside_grant`,
`task_outside_grant`. Telemetry routes these to Tel-2 governance sink.

## Telemetry

When a `TelemetryEmitter` is wired, the broker emits:

- `capability.grant.issued` — `info`
- `capability.grant.revoked` — `info`
- `capability.check.allowed` — `debug`
- `capability.check.denied` — `warn`

Attributes carry only `kind`, `subjectKind`, `subjectId`, and (on
denial) `reason`. The grant id, raw constraint payloads, and the
caller-supplied `reason` are NOT emitted. Tel-0's auto-scrub catches
trivial leaks (e.g. `password=hunter2` in a reason); the tests prove
that boundary.

## Tests (16)

- grant + listFor + revoke (3)
- no-grant denial (1)
- require throws CapabilityDenied with code "capability.denied" (1)
- wildcard grant matches any resource (1)
- pathPrefixes match + reject + reject-on-absent (2)
- host exact match + reject (1)
- provider/connector/workspace/task list membership (1)
- expired grants reported as "expired" + filtered from listFor (2)
- revoke idempotent + revokeSubject drops every grant (2)
- telemetry routing for issued / allowed / denied / revoked (1)
- telemetry never leaks the grant `reason` field (1)
- CAPABILITY_KINDS has all 13 entries (1)
- require returns the grant on success (1)

## Wow moment

The "telemetry never leaks the grant `reason`" test plants a
`password=hunter2` string in the grant's `reason` and asserts the
emitted event JSON does NOT contain it — proving the boundary
protects sensitive data even when the caller volunteers it.

## Handoff

D-2 (main-process IPC migration) wraps every privileged handler with
`broker.require({ subject, kind, resource })`. The agent runtime
(A-2) holds a broker reference and calls `require` before each tool
invocation that maps to a capability kind. Cells receive a scoped
broker (no `grant` access) through the SDK (Cell-3).
